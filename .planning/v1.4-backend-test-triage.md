### spawn-assignment-handler

- **Path:** `src/__tests__/spawn-assignment-handler.test.ts`
- **Verdict:** stale-assertion (ambiguous — user-confirmed via `auto-assign-all`)
- **Evidence:** 27/28 pass. The "60 tight-loop requests" test (L277-283)
  expects `accepted >= 45 && limited > 0`. Got `accepted=0, limited=0`.
  After the PDP refactor at commit `5ce8d59 refactor(handlers): route
  reverse-RPC handlers through the PDP (PDP is sole gate)`, the test's
  `makeCtx()` defaults `maxPerHour: 10, maxConcurrent: 3` (L117) cause
  EVERY one of the 60 requests to be perm-denied by the PDP before the
  rate-limit ever fires (limit `maxPerHour=10` but tight-loop sends 60,
  so attempts beyond 10 hit the spawnAgents.maxPerHour permission check
  rather than the JSON-RPC rate-limit error code -32029). The handler
  returns -32000 (perm-denied) not -32029 (rate-limited), so neither
  `accepted` nor `limited` increments. The test was calibrated against
  the pre-PDP world where the rate-limit ceiling, not the per-grant
  permission ceiling, gated tight-loop behaviour.
  ```
  expect(received).toBeGreaterThan(expected)
  Expected: > 0
  Received: 0
  ```
- **Action:** Bump `makeCtx()` defaults at L117 from
  `{ maxPerHour: 10, maxConcurrent: 3 }` to
  `{ maxPerHour: 1000, maxConcurrent: 100 }` so the rate-limiter
  ceiling (not the PDP) gates the 60-tight-loop test. The 5 tests that
  explicitly need lower ceilings (L214, L339, L354, L372) pass their
  own `grantedPermissions` overrides, so they remain unaffected.
