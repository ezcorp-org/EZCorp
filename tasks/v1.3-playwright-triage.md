They are **test debt**: v1.3 changed 116 source files under `web/src/` (38 routes + 25 components + 3479 net inserted component lines per `git diff --stat b7c2e26..HEAD`) without updating the e2e specs to match. Of the 14 spec files that v1.3 *did* touch, 13 pass; only `accessibility-mobile.spec.ts` fails, and that failure surfaces an actual a11y bug introduced by v1.3.

Release recommendation: **safe to ship v1.3** on the strength of the 187/187 backend integration + 35/35 vitest + 13/14 v1.3-touched Playwright passes. The 399 stale-locator failures are a Playwright-suite hygiene debt that v1.4 should pay down (a candidate area `tasks/v1.4-e2e-coverage-100.md` already exists). One concrete release blocker found: see Section A item A1.

## Section A — v1.3-introduced failures (must fix before release: 1)

Triage rule: a failure is "v1.3-introduced" only if **(a) the spec file or its SUT was modified after `b7c2e26` (v1.2 ship)** AND **(b) the failure is a real product bug, not a stale assertion**.

By that rule, exactly **one** failure qualifies as a release blocker:

### A1. `accessibility-mobile.spec.ts:82` — Phase 49.5 mobile a11y, real WCAG violations on `/agents/mobile-agent`

- **Spec:** `web/e2e/accessibility-mobile.spec.ts:82:3`
- **Test:** `Phase 49.5 — WCAG 2.1 AA @ 375×667 (mobile) > Agent Detail (/agents/mobile-agent) — no a11y violations on mobile`
- **Spec file modified in v1.3?** **YES** — file is part of the v1.3-touched set (`git diff --name-only b7c2e26..HEAD -- web/e2e/`).
- **Error excerpt:**
  ```
  Error: Mobile a11y violations on Agent Detail:
  [serious] color-contrast: contrast ratio thresholds
    Affected: <button class="rounded-md bg-amber-600 px-4 py-2 text-sm font-medium text-white …">
  [critical] label: Ensure every form element has a label
  ```
- **Why v1.3:** Phase 49.5 added the mobile-axe scanner; the SUT (`/agents/[name]/+page.svelte`) was modified in v1.3 (commits 8383a2a `feat(agents): fuzzy search …`, plus the wider Phase 49 navigation pass touched the layout/sidebar — `13fe28d feat(layout): mobile-responsive sidebar with SwipeDrawer at <lg breakpoint`). The amber CTA button and an unlabeled form input on Agent Detail were never run through axe at 375x667 before.
- **Recommended action:** Either (a) fix the two violations on Agent Detail (add an `aria-label` to the form element, swap `bg-amber-600 text-white` for a contrast-passing variant), or (b) explicitly accept-list the violations with a comment + ticket. Treat as **release blocker** — Phase 49.5 was scoped specifically to enforce mobile WCAG 2.1 AA.

## Section E — Infra issues / unclassified

### E1. `validate-prod-shape.spec.ts` (5 fails) — hard-coded `localhost:5173`

- All 5 failures in this file are `Error: page.goto: net::ERR_CONNECTION_REFUSED at http://localhost:5173/…`.
- The Playwright config points at `localhost:4173` (vite preview); these tests bypass the fixture and hard-code the dev-mode port `5173`.
- This was failing before v1.3 too (the spec is unchanged; the misconfiguration is in the spec itself).
- **Action:** v1.4 — either start a dev server alongside preview for these specs, or refactor to use `baseURL`. Don't block v1.3 release.

### E2. `[WebServer] server error: status 500 — "Database not initialized — call initDb() first"` log spam

- ~680 occurrences of `"server error","subsystem":"hooks.server","status":500` in the run log. **NOT** a test failure; these are SvelteKit server-side errors emitted while tests run with `PI_SKIP_INIT=1`.
- This is **by design**: `web/src/hooks.server.ts:22` skips `ensureInitialized()` when `PI_SKIP_INIT` is set, and the e2e fixtures intercept all API calls at the Playwright route layer. The 500s are legitimate "DB not initialized" responses for any request the mock didn't intercept (e.g., SSR data loaders, prerender hits, idle-tab background fetches).
- **Action:** consider gating the log to `level=debug` when `PI_SKIP_INIT` is set, but this is purely a log-noise concern, not a release blocker.

### E3. Possible flake — 1 test only `agent-chat.spec.ts:32` exceeded the full 30s timeout

- `Chat button creates conversation and navigates to chat (30.0s)` — locator never resolved. The earlier `agent-chat.spec.ts:5` test fails with the same ` Chat with this agent ` button-not-found pattern, suggesting a deterministic UI/mock mismatch rather than a flake. Roll into Section B's chat-page consumer category for v1.4.

### E4. Worker count

The run executed with default workers (no `--workers=N` override). On this machine with `4` cores, `parallelIndex` 0..3 was active (i.e. ~4-way parallel by default per `playwright.config.ts:11`'s `workers: process.env.CI ? 4 : undefined`). Re-running with `--workers=1` may resolve a small number of cross-test interference cases, but spot-checks in the log show no clustered timeouts that would indicate parallelism damage. Not investigated further.
