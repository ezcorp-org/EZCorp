## Claim 1 — PDP fail-closed when unwired (C6 in design)

**Verdict:** PASS, with a MEDIUM caveat.

**Evidence:**
- `src/extensions/permission-engine.ts:128-134` — `createPermissionEngine` throws when `registry`, `bus`, or `db` is missing.
- `src/extensions/permission-engine.ts:309-320` — `getPermissionEngine()` throws `"PermissionEngine not initialized — first call must provide deps"` when called with no deps before init.
- `web/src/lib/server/context.ts:138-171` — boot path constructs the singleton with real deps inside a try/catch; the catch only wraps `bootSpawnFlaggedBundledExtensions`, so a successful `getPermissionEngine({...})` call DOES set the singleton even if boot-spawn later errors.
- `src/runtime/executor.ts:380-384` — agent-run hot path also re-initializes the engine with concrete deps (idempotent — the singleton wins).
- Test coverage: `src/__tests__/permission-engine.test.ts:95-125` — three explicit tests assert throw on missing registry/bus/db.

**Reasoning:** The factory enforces deps strictly, all production call sites either pass deps or rely on the pre-init singleton, and there is no caller that catches the "not initialized" error and converts it to `allow`. Errors propagate up through SvelteKit's standard error handler (HTTP 500) — never an implicit allow.

**MEDIUM caveat (separate finding):** `web/src/routes/api/extensions/[name]/events/[event]/+server.ts:313-330` catches the engine call inside a try/catch around `getPermissionEngine() + ToolExecutor + ensureSubprocessRpcWired`, marks `wireOk = false`, and **continues to call `handleAppendMessageRpc` at line 372 with a manually-built ctx that omits the engine field**. Inside `src/extensions/append-message-handler.ts:197-215`, when `ctx.engine` is falsy the handler falls back to the legacy `granted.appendMessages` boolean check. This is NOT a fail-open (the legacy boolean still gates the call), but the PDP audit row is never written, scope checks (`alwaysAllow`) are never applied, and the per-conversation override at `permission-engine.ts:191-198` is bypassed for messageToolbar events. Audit visibility for this path is broken.

**Recommendation:** keep the verdict as PASS for the strict claim ("does any path return allow when engine isn't there?"). Fix the audit-visibility gap by passing `engine: getPermissionEngine()` into the `AppendMessageContext` for the messageToolbar shortcut so the PDP fires + audits even when subprocess wiring fails.

## Claim 2 — Manifest ceiling enforcement on reapprove

**Verdict:** HIGH.

**Evidence:**
- `web/src/routes/api/extensions/[id]/reapprove/+server.ts:115-146` — handler reads `ext.manifest?.permissions` (line 125-126) and writes it verbatim into `grantedPermissions[grantKey]` (line 141). No clamp against `clampExtensionPermissions`. No clamp against `clampToBundledCeiling`. No comparison against the prior `grantedPermissions` value. No record of the user's install-time narrowed choice.
- `tasks/capability-expiry-design.md:444-446` — §9 claims "the default reapprove path is **bounded by the manifest ceiling**". This is *technically* true for non-bundled extensions (the manifest IS the ceiling). For bundled extensions the manifest can declare more than `BUNDLED_CEILING` allows; the ceiling is enforced at install in `src/extensions/bundled.ts:700-710`, but reapprove skips it entirely.
- `.planning/milestones/v1.3-MILESTONE-AUDIT.md:18-19` — the milestone audit acknowledges this as v1.5 tech debt: *"Restoring install-time narrowed choices on re-approve (currently re-grants full manifest value) — flagged for v1.5"*.
- The `scope` body parameter (`reapprove/+server.ts:108-113`) is purely an admin gate; it does NOT translate to anything stored on `grantedPermissions`. There is no per-grant scope field on the extensions row.

**Reasoning:** The actual contract enforced is weaker than the §9 promise. Concrete escalation paths:
1. **Bundled-ceiling bypass:** if a bundled extension's `manifest.permissions` exceeds `BUNDLED_CEILING[name]` (e.g. tampering or a buggy manifest update), the reapprove POST writes the manifest claim, not the ceiling. The `manifest.lock.json` checksum gate would catch tampering at boot, but this is single-defense; reapprove should also clamp to the ceiling.
2. **Lost user narrowing:** if a user originally approved a NARROWED subset at install (via `clampExtensionPermissions` accepting `submittedPerms` smaller than `manifestPerms`), the reapprove silently restores the FULL manifest value. From the user's perspective this is an escalation; from §9's perspective it's "within the manifest ceiling" so technically not an escalation. The user-visible behavior breaks the principle of least surprise.
3. **No scope down-clamp:** if the original always-allow was `session`-scoped (by design, expires on restart), reapprove on the *extension* row writes a `forever`-equivalent grant directly on `grantedPermissions` (no expiry until next sweep). Always-allow rows are stored separately and reapprove doesn't touch them, so there is no symmetric path that respects the original scope.

**Exploit / repro:** install a bundled extension whose manifest declares `network: ["api.foo.com", "api.evil.com"]` but `BUNDLED_CEILING` allows only `["api.foo.com"]`. The install-time clamp at `bundled.ts:700-710` writes `grantedPermissions.network = ["api.foo.com"]` correctly. Wait for the capability-expiry sweep to revoke `network`. POST to `/api/extensions/<id>/reapprove` with `{capability: "network"}`. Result: `grantedPermissions.network = ["api.foo.com", "api.evil.com"]` — the bundled-ceiling has been silently bypassed.

**Recommendation:** in `reapprove/+server.ts`, replace the line-141 raw write with a call to `clampExtensionPermissions` against `manifestPerms` AND, when the extension is bundled, against `getCeiling(extensionName)`. Additionally, persist the install-time narrowed choice on the extensions row (e.g. `installedPermissions: ExtensionPermissions`) so reapprove can restore the user's narrowed grant rather than the full manifest.

## Claim 3 — Cross-extension capability intersection on `ezcorp/invoke` (C1 in design)

**Verdict:** HIGH.

**Evidence:**
- `src/extensions/tool-executor.ts:946-959` — intersection only computed if `calleeGrants.acceptsCallerCaps === true`:
  ```ts
  let capContext: CapabilitySet | undefined;
  if (calleeGrants?.acceptsCallerCaps === true) {
    const callerCaps = upstreamRuntimeCtx?.currentCapContext ?? grantsToCapabilitySet(...callerExtId);
    const calleeCaps = grantsToCapabilitySet(calleeGrants);
    capContext = intersect(callerCaps, calleeCaps);
  }
  ```
- `src/extensions/tool-executor.ts:973-981` — when `acceptsCallerCaps !== true`, `capContext` is undefined and `executeToolCall` runs without it.
- `src/extensions/permission-engine.ts:187-198` — when `ctx.capContext` is undefined, the PDP falls back to the callee's full installed grants from the registry. No intersection.
- `src/__tests__/cross-ext-attribution.test.ts:166-201` — tests (a) and (b) explicitly assert the default behavior is *no intersection*: *"acceptsCallerCaps absent → no intersection, callee runs with own grants"*.
- `.planning/milestones/v1.3-permission-system-original-plan.md:27` — design pillar 7 says: *"Closes confused-deputy finding C1. handlePiInvoke computes effectiveCaps = intersect(callerCaps, calleeCaps)"* with no qualifier. The doc reads as "intersection by default".

**Reasoning:** The implementation inverts the design's stated default. The design said intersection IS the default (deputies opt out via `acceptsCallerCaps: true` to keep their own caps). The code makes intersection OPT-IN — only callees explicitly marked `acceptsCallerCaps: true` get the intersection treatment. Practical impact: a confused-deputy attack (A with no network → B with broad network → A invokes B with malicious URL) is **NOT prevented by intersection**; it's prevented only by whatever per-call gate the callee already had. For most extensions there is no such gate.

The implementation IS internally consistent: in cross-ext-attribution tests (c)–(i) where `acceptsCallerCaps: true` is set, the intersection is computed correctly (including the M1 chained-deputy fix at `tool-executor.ts:920-945`). The integration test at line 706 explicitly named *"CONFUSED-DEPUTY integration"* tests only the case where the callee IS a marked deputy. There is no test for the case where a non-deputy callee is invoked by a less-privileged caller — because by design that case is treated as *not a confused deputy*.

**Exploit / repro:**
1. Install extension A with `permissions: {}` (no network).
2. Install extension B with `permissions: { network: ["api.foo.com", "api.attacker.com"] }` and DO NOT mark `acceptsCallerCaps: true`. (Most legitimate extensions would not declare this flag — it's bundled-deputy-only ergonomics.)
3. B exposes a tool `proxyFetch(url: string) → fetches url`.
4. A invokes `B__proxyFetch` with `url: "api.attacker.com"`.
5. PDP authorizes against B's installed grants → `network:api.attacker.com` is granted to B → call succeeds.
6. A has effectively gained network access through B.

**Recommendation:** make intersection the default, as the design specified. Concretely: in `handlePiInvoke`, ALWAYS set `capContext = intersect(callerCaps, calleeCaps)` and treat `acceptsCallerCaps: true` as the OPT-OUT (caller's caps not used; callee runs with full installed grants). This re-aligns the code with the original design's "deputies opt out via `acceptsCallerCaps: true` declaration" wording. Update tests (a) and (b) accordingly. If keeping the current opt-in semantics is intentional, the design doc and §C1 audit-finding closure must be updated to reflect that the codebase relies on EXPLICIT `acceptsCallerCaps: true` markings on every deputy — and a sweep of all 9 bundled extensions should confirm none ship as confused-deputy targets without the flag.

## Claim 4 — MCP isolation depth

**Verdict:** PASS for the stated today-story, with KNOWN GAPS that are documented in code.

**Evidence:**
- `src/extensions/mcp-sandbox.ts:68-188` — `buildSandboxedMcpSpec` wraps the MCP spawn with `prlimit` (memory bounds) + `unshare -U -m` (user + mount namespace, no net) on Linux. `HTTPS_PROXY` env injected with per-instance bearer-token URL.
- `src/extensions/mcp-netns.ts:6-39` — file-level docstring is honest: *"Network isolation is intentionally NOT done via `-n` … Per-host network enforcement is done by the forward proxy on host loopback"*.
- `src/extensions/mcp-launcher.sh:1-50` — launcher exec'd inside the user+mount namespace; attempts `cap_sys_admin` drop via `capsh` but falls through to `exec "$@"` if `capsh` unavailable. Comments at line 36-38 acknowledge the cap-drop is best-effort.
- `src/extensions/mcp-proxy.ts:115-122` — `createMcpProxy` throws if `engine` missing (fail-closed at the proxy gate too).
- `src/extensions/mcp-proxy.ts:278-293` — bearer-token compare uses `crypto.timingSafeEqual` with length-equality short-circuit. No timing oracle.
- `src/extensions/mcp-proxy.ts:295-335` — internal-host hard-deny + per-host PDP authorize on every CONNECT. Internal-host check uses `src/extensions/runtime/internal-host.ts:34-58` (single shared regex with normalization).
- `.planning/milestones/v1.3-MILESTONE-AUDIT.md:39` — explicit deferral: *"Kernel-level network isolation for MCP (Phase 7 dropped -n from unshare; today = bearer-token proxy + prlimit + user/mount namespace)"*.

**Reasoning:** The implementation matches the brief's today-story exactly: bearer-token proxy on host loopback + per-host PDP gate + user/mount namespace + prlimit. No kernel-level netns. The gaps are openly documented:

1. **No netns:** the MCP shares the host's network namespace. A determined MCP can `Bun.connect` directly to 127.0.0.1:* and reach any host-loopback service. The proxy's bearer-token IS scoped per-MCP and the internal-host regex blocks RFC-1918/loopback at the proxy gate, but that's *only* enforced if the MCP uses `HTTPS_PROXY`. A raw socket via `socket(AF_INET, SOCK_STREAM)` to 127.0.0.1 bypasses the proxy entirely. **Not defended against.**
2. **Token in env:** `HTTPS_PROXY=http://_:<token>@127.0.0.1:<port>` is in the MCP's process environ. Any same-user host process that can read `/proc/<pid>/environ` (default `0644` on Linux for the owner) can steal the token. Not exploitable cross-user (the env file is mode 0600 for non-owner readers), but worth noting.
3. **`/tmp` not isolated:** the mount namespace gives a fresh mount table but no `tmpfs` or bind-remount of `/tmp`. The MCP sees the host's `/tmp` and can side-channel through it.
4. **`ptrace` not blocked:** no seccomp profile applied. Same-uid processes can ptrace the MCP and vice versa.
5. **DNS rebinding:** `isInternalHost` matches against the URL hostname literal, not the resolved IP. A public DNS name resolving to `127.0.0.1` (rebinding attack) would bypass the internal-host gate at the proxy. The comment at `internal-host.ts:30-31` acknowledges this — *"if it lands on a private IP, the host's PDP/network stack governs (Phase 7 adds the kernel-level netns gate for that case)"* — but the netns gate was dropped, so the comment is now stale.
6. **`cap_sys_admin` drop is best-effort:** `mcp-launcher.sh:42-50` falls through to direct exec when `capsh` is missing. On the slim Bun image the package is included; on user-installed Linux fallbacks it may not be.

**Reasoning continued:** the brief explicitly framed the today-story as *"NO kernel-level netns; instead user/mount namespace + prlimit + bearer-token proxy + per-host PDP gate"*. The code does exactly that. Verdict is PASS for the stated bar.

**Recommendation:**
- Add a `127.0.0.1` regex case to `INTERNAL_HOST_RE` that ALSO catches public DNS names resolving to private IPs at proxy CONNECT time (resolve via `Bun.dns.lookup`, then re-check `isInternalHost(resolvedIp)`).
- Update `internal-host.ts:30-31` comment to match the post-fix-pass reality: drop the *"Phase 7 adds the kernel-level netns gate"* line and replace with *"resolved IP is not validated; DNS rebinding is a known gap deferred to a future hardening phase"*.
- Document the env-leak surface for the bearer token in deployment docs; recommend `EZCORP_MCP_PROXY_USE_HEADER=1` mode (CONNECT with a `Proxy-Authorization` header set inside the MCP process at connect time, not in env) for high-trust deployments.
- Plan a Phase 8 hardening pass: add seccomp + tmpfs `/tmp` overlay + restore netns with the veth-pair workaround. The reasons to defer (UDS unparseable, complexity) are real, but the net of the today-story is "the MCP is a same-host trust peer", which is a wider surface than the design intent.

## Cross-cutting findings

### CC1 — `loadConversationOverride` swallows DB errors and returns null (LOW)
`src/extensions/permission-engine.ts:382-386` — `try { await getConversationExtensionEffectiveGrants(...) } catch { return null }`. The comment at lines 365-369 frames this as fail-closed: *"a DB blip can't fail-open the PDP — the caller falls back to the registry grants on null"*. But the registry grants may be WIDER than the conversation override (the override is the intersection of parent and child agent caps for spawned conversations — `permission-engine.ts:184-186`). A DB blip during a child-conversation tool call therefore SILENTLY WIDENS the cap envelope back to the registry's full grant. This is fail-OPEN relative to the intended sub-conversation cap inheritance from §H3.

**Recommendation:** on DB failure, prefer "deny" with reason `"override-lookup-failed"` rather than silent fallback. Or: cache the override at conversation creation time so a transient DB outage doesn't widen the envelope.

### CC2 — `writeAuditRow` swallows audit-write failures (LOW)
`src/extensions/permission-engine.ts:545-548` — *"Audit write failures must never block a tool call. The PDP's primary job is the decision; logging is best-effort."* Reasonable for availability, but the audit log is a compliance artifact. There should be a fallback path (e.g. structured stderr log with action+extensionId+capability) so a missing audit row is observable in the host logs even when the DB write fails.

### CC3 — `MAX_CALL_DEPTH = 10` is per-chain, not per-conversation (LOW)
`src/extensions/tool-executor.ts:843` — depth is taken from `params._depth` provided by the caller; the recursion in `handlePiInvoke` increments at line 980 with `_callDepth: depth + 1`. A malicious extension could craft 10 invoke chains in parallel rather than serially and bypass the cap. Not a single-thread escape, but a fan-out cost surface. Combine with the per-extension rate limiter for production.

### CC4 — Reapprove writes audit row with action `PERMISSION_GRANTED`, not a dedicated reapprove action (LOW)
`web/src/routes/api/extensions/[id]/reapprove/+server.ts:154` — uses `EXT_AUDIT_ACTIONS.PERMISSION_GRANTED` and disambiguates via `metadata.reason: "user-reapprove"`. This is searchable but couples reapproves to first-time grants in the audit stream. A dedicated `PERMISSION_REAPPROVED` action would simplify SOC 2 / SIEM integrations.

### CC5 — `internal-host.ts` regex `^localhost` is unanchored at the end (NIT)
`src/extensions/runtime/internal-host.ts:34` — `/^(localhost|127\.|...|::1$)/i`. Only `::1` has end-anchor. `localhost.evil.com` would match the `^localhost` branch and be (incorrectly) blocked as internal — false positive, not a security hole. `127foo.com` would similarly not match (the `\.` enforces the dot). Defensive overall, but worth noting that the regex's anchoring is asymmetric.
