## Item 4 — Production deployment dry-run

**Note:** prod Dockerfile exposes **port 3000**, not 4173. (The original prompt said 4173 — that's `vite preview`. Production uses `bun run web/build/index.js` on 3000.)

### 4.1 — Build the image
```bash
docker build -t ezcorp-v1.3-rc .
```
Expected: clean build, final image tag `ezcorp-v1.3-rc`. If build fails on workspace `bun install`, the issue likely predates v1.3 — capture stderr + bail.

### 4.2 — Boot with empty PGlite
```bash
DATA=$(mktemp -d)
docker run --rm -d --name ezcorp-rc -p 3000:3000 -v "$DATA:/data" ezcorp-v1.3-rc
echo "Data dir: $DATA"
```
Wait ~10–15s for migrate() to complete on the empty DB.

### 4.3 — Liveness + readiness
```bash
curl -s -w "\n%{http_code}\n" http://localhost:3000/api/health
curl -s -w "\n%{http_code}\n" http://localhost:3000/api/ready
```
Expected: both `200`. `/api/ready` body should contain `"state":"ready"`. If `503`, migrate() didn't finish or failed — capture body + check container logs:
```bash
docker logs ezcorp-rc 2>&1 | tail -50
```

### 4.4 — Smoke flow (browser, http://localhost:3000)
1. Sign up a new user (whatever the signup flow is — check `web/src/routes/(auth)`)
2. Install a built-in extension (likely from `/extensions` or `/marketplace`)
3. Fire one tool call from a chat (any extension that triggers a capability event pill)
4. Open `/audit` — verify the call shows up in the 24h strip and the per-extension drill-down

Report each as `✅` / `❌ <what broke>`.

### 4.5 — Migrate-failure rollback verification
This validates the rollback path at `src/db/connection.ts:125`. Approach:

```bash
# Stop the container cleanly
docker stop ezcorp-rc 2>/dev/null

# Corrupt the data dir's PGlite state to force migrate() failure on next boot
# (PGlite stores files under <DATA>/base/<oid>/...; truncating one file is enough)
ls "$DATA"
# Pick any file inside the data dir and truncate it:
find "$DATA" -type f -name "*.dat" 2>/dev/null | head -3
# CAREFUL: only do this on the temp $DATA dir, never on real data.
# If you'd rather not corrupt, skip 4.5 — it's a confidence build, not a release blocker.
```

If you proceed with corruption, expect:
- Container boots, migrate() fails
- `rollbackMigration` runs: `<DATA>` renamed to `<DATA>.failed.<ts>`, latest snapshot restored
- Container exits(1); Docker restarts it; circuit breaker skips migrate() second time
- `docker logs` should show `"Restored pre-boot snapshot"` and `"circuit breaker skip"` lines

If no snapshot exists yet (fresh empty DB has none), this won't trigger the restore path — need an existing DB with a snapshot to test it. Easier path: defer this to a real pre-prod environment with backups.

### 4.6 — Cleanup
```bash
docker stop ezcorp-rc 2>/dev/null
rm -rf "$DATA" "$DATA".failed.* 2>/dev/null
```

## Item 5 — Manual UAT against the security backbone

Run against a freshly-booted container OR your local dev — but **not while running tests** (PGlite single-writer).

### 5.1 — 4-scope capability-grant modal on extension install
1. Go to `/extensions` (or marketplace), pick a built-in that declares capabilities (e.g., one that uses `net:fetch` or `storage:read`)
2. Click "Install"
3. Expect: a modal appears showing the extension's requested capabilities and a 4-radio choice: **session / conversation / project / forever**
4. Verify "forever" is admin-gated (disabled if you're not admin, or only shown to admin role)
5. Pick "session" → install completes → grant is created with session scope
6. **❌ if:** modal doesn't appear, or all 4 options are equally pickable for non-admin users, or extension installs without prompting at all

### 5.2 — Hourly HostMaintenanceDaemon sweep + ExpiredGrantsBanner
Manual sweep (don't wait an hour):
```bash
bun run scripts/sweep-perm-expiry.ts
```
This forces the sweep to mark expired grants. Then:
1. Refresh the settings page (or wherever `ExpiredGrantsBanner.svelte` is mounted — likely settings/permissions tab)
2. Expect: a banner appears listing the grant(s) you just expired (or any that were already expired)
3. **❌ if:** banner doesn't render, banner shows wrong extension/capability, or sweep script errors

### 5.3 — "Re-approve" manifest-ceiling clamp
1. From the ExpiredGrantsBanner, click "Re-approve" on a grant whose original install-time scope was, say, `session`
2. The re-approve modal opens (`ExpiredReapproveModal.svelte`)
3. Try to pick **"forever"** as the new scope (or whatever scope is broader than the install-time ceiling)
4. Expect: either the broader option is disabled/hidden, OR the request is rejected server-side with a clamp message
5. **❌ if:** the API returns success and the new grant has scope=forever despite install-time being session (this is the C2 / capability-expiry-design §9 contract — a real security bug if it fails)

### 5.4 — In-chat capability event pills (built-in extensions)
1. Open any project chat
2. Send a message that triggers a built-in extension's tool call (e.g., a search or memory write)
3. Expect: a capability event pill renders inline in the assistant turn — "lessons-distiller used `storage:write`" or similar
4. Click the pill → expect a small drill-down (which capability, target, decision)
5. **❌ if:** no pill renders for built-in extensions, or pill renders but data is wrong/empty

### 5.5 — `/audit` admin page
1. Navigate to `/audit` (route: `web/src/routes/(app)/audit/+page.svelte`)
2. Expect: 24h stats strip at top (calls accepted/denied, top extensions)
3. Click into a per-extension row — expect a drill-down list of recent calls with: timestamp, capability, target, scope, decision
4. **❌ if:** stats are obviously wrong (e.g., zero when you just made calls), drill-down 500s, or extension link is broken

### 5.6 — Manual full expiry → chat re-prompt
1. Pick one extension you've granted to and have an open chat with
2. From settings or via a direct API call, manually expire ALL its grants (you can use the same sweep script or hit `/api/permissions/expire?extensionId=<id>` if such an endpoint exists)
3. Send a chat message that would invoke that extension's tool
4. Expect: the chat surfaces a re-prompt modal asking you to re-approve before the tool runs
5. **❌ if:** the tool call silently fails with no re-prompt, OR runs successfully despite expired grants (security bug)
